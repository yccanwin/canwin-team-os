# CanWin Team OS 3.0 销售工作台最终开工方案

> 文档状态：开工基线
> 适用项目：CanWin Team OS 3.0
> 技术底座：React + TypeScript + Vite + Supabase + GitHub Pages
> 时区：Asia/Shanghai（北京时间）
> 原则：增量建设，不重写 2.0；安全底座未验收前，不录入真实客户数据。

## 1. 项目目标

把客如云代理商的销售与交付经验固化为可执行、可追踪、可审计的工作流，使团队在老板不参与日常盯单时仍能稳定运转。

3.0 首期只打通以下完整闭环：

```text
线索 → 商机 → 报价 → 定金 → 内部订单 → 软件/硬件履约
     → 安装培训 → 售后交接 → 续费或结束
```

成功标准不是“页面完成”，而是一个真实订单能够按权限、规则、财务和交付要求从头跑到尾，所有关键变化都有记录。

## 2. 范围边界

### 2.1 本期必须完成

- 管理员邀请账号和多角色权限。
- 品牌客户、多门店、联系人、线索、公海、商机和跟进。

- 产品、套餐、加购项、报价和报价版本。
- 定金、客户付款、销售汇款、内部采购款、价差和退款冲销。
- 一张订单内的软件、硬件两套履约状态。
- 库存扣减、缺货异常、预计到货日。
- 安装培训、售后群、运维接手和续费。
- 手机端“今日”行动队列。
- 主管异常板、销售会议板、季度目标和官方对账。
- 企业微信摘要、预约提醒和失败重试。

- 个人费用报销。
- 从客如云后台自动采集数据。
- 小程序、菜品图和企业运营模块。

## 3. 固定业务口径

### 3.1 客户层级

```text
品牌客户（总部）
  ├─ 门店 A
  ├─ 门店 B
  └─ 门店 C
```

- 直营连锁：总部销售为品牌负责人，可创建一张主订单并分配多个门店交付。
- 加盟门店：默认独立归属、独立商机、独立报价和独立订单。
- 同一联系人可以关联品牌和多个门店，但必须指定主要关系。
- 品牌、门店、联系人都保留合并和去重记录，不做物理删除。

### 3.2 A/B/C/D 分级

等级只表达业务价值和交付复杂度，不表达成交意向。

系统按 `A > B > C > D` 的优先级自动计算：

- A：连锁、宴会或主管确认的标志性门店。
- B：面积大于等于 300㎡，或包厢大于等于 5 个。
- C：适合年费产品，但不满足 A/B。
- D：纯外卖店，或明确不适合年费产品。

补充规则：

- “标志性门店”由销售提出、主管确认。
- “适合年费产品”通过固定选择项记录，不允许只填自由文本。
- A 类商机未完成演示，不允许确认定金。
- D 类退出有效漏斗，可进入长期培育，但不计有效商机。

### 3.3 有效商机

同时满足以下条件才进入有效漏斗：

1. 存在真实门店名称和经营地址。
2. 等级为 A、B 或 C。
3. 适合年费产品。
4. 已接触关键人，或已明确约到关键人。

系统自动计算结果；销售不能直接修改“是否有效商机”。基础事实变化后重新计算，并记录变化原因。

### 3.4 跟进有效性

有效跟进必须同时包含：

- 新获得的业务事实或客户明确承诺；
- 下一步动作；
- 下一步时间。

未接电话只记录为联系尝试：

- 计入首次联系动作；
- 不重置 15 天无有效跟进计时；
- 三个不同自然日均未联系到，进入首轮 30 天培育；
- 第二次进入培育需主管审核。

### 3.5 公海与回收

- 区县公海相互隔离。
- 销售只能领取自己有权限区域的线索。
- 领取不限数量，但领取必须通过数据库事务，防止多人同时领取。
- 已占用线索对其他销售只显示“已被领取”和负责人姓名，不显示电话及跟进内容。
- 新线索 24 小时未产生联系尝试时标红。
- 新线索 48 小时未产生联系尝试时自动回收。
- 已领取线索 15 天无有效跟进时自动回收。

以下情况暂停自动回收：

- 已确认未来预约，且预约尚未逾期；
- 已报价且决策日期未到；
- 已确认定金；
- 负责人请假且已设置临时代理；
- 主管设置了有截止时间的暂停回收原因。

暂停结束后继续计算剩余时间，不重新获得完整周期。

### 3.6 新店与竞品老店节点

新店倒排：

- A/B：开业前 30 天确认定金、15 天下内部订单、3 天完成安装培训。
- C：开业前 15 天确认定金、7 天下内部订单、2 天完成安装培训。

竞品老店：

- 到期前 60 天完成情况摸底。
- 到期前 30 天完成方案和报价。
- 到期前 15 天确认决策，或记录结构化失败原因。

开业日或到期日变化时：

- 只重新计算未完成节点；
- 已完成节点不改变；
- 保存原日期、新日期、修改人、修改时间和修改原因。

### 3.7 报价规则

- 产品、硬件、积分、套餐和加购目录按业态维护。
- 每次目录发布生成不可修改的目录版本。
- 套餐内容固定；销售只能调整允许销售调整的客户价格。
- 改动套餐内容、低于最低价或其他特殊条款，必须提交主管审批。
- 报价默认有效 15 个自然日。
- 每次保存报价都生成版本；界面默认只显示当前版本，不物理覆盖旧版本。
- 定金确认时冻结最终报价快照。
- 定金后的增减内容通过报价变更单处理，原报价不修改。

内部结算价：

```text
内部结算价 = 冻结采购成本 × 1.10
```

- 计算结果以分为最小单位四舍五入。
- 成本、加成比例和计算结果随订单冻结。
- 后续目录改价不影响历史订单。

### 3.8 付款与财务

系统区分四类资金事件：

1. 客户付款：客户支付给公司或销售。
2. 销售汇款：销售把代收款汇给公司。
3. 内部采购款：销售应结清的内部订单金额。
4. 价差支付：公司最终确认应付销售的价差。

统一规则：

- 所有资金事件先登记为“待确认”。
- 只有财务可以确认、驳回或冲销。
- 财务确认定金后，订单记为销售成交。
- 客户全款、退款和内部采购款都完成后，才能确认最终价差。
- 销售未结清内部采购款，公司不得开始软件开通和硬件备货。
- 已确认的付款不能编辑或删除，只能新增等额或部分冲销记录。
- 每笔资金记录必须包含付款方、收款方、金额、方式、时间、关联订单和凭证状态。
- 公司收款、销售代收采用同一台账，但必须明确 `recipient_type`。

价差口径：

```text
预计价差 = 冻结客户成交价 - 冻结内部结算价 - 已知销售承担费用
最终价差 = 财务确认的客户净收款 - 内部结算价 - 退款 - 销售承担费用
```

公司利润与销售价差是两套口径，不互相替代。

### 3.9 订单、库存与交付

- 一个客户订单只有一个订单号。
- 订单内部同时保留软件履约和硬件履约状态，不拆成两张订单。
- 连锁主订单通过门店交付明细分配数量、状态和负责人。
- 库存只记录 SKU 和数量，不记录硬件序列号。
- 库存不足时不允许产生负库存；生成缺货异常和预计到货日。
- 库存预留、实际出库和取消释放必须使用数据库事务。
- 定金确认后 24 小时内必须创建售后群并发起标准交接。
- 销售填写交接清单，线上运维确认接手。
- 实施人员完成“安装”和“培训”两项即可完成实施。
- 不强制客户电子签名，不要求重复上传照片；允许记录实施说明和异常。

### 3.10 续费

- 软件开通或历史产品录入时必须保存到期日。
- 到期前 60/30/15 天自动生成续费节点。
- 连锁品牌支持按品牌批量创建续费任务，但每家门店保留独立结果。
- 续费 GMV 与新签 GMV 分开统计。

## 4. 用户、角色与权限

### 4.1 账号规则

- 关闭公开注册。
- 账号只能由管理员邀请或创建。
- 离职时先停用账号，再由主管批量转移客户、商机、任务和订单。
- 停用账号不能登录，但历史记录中的负责人不被删除。
- 请假不转移所有权，使用有起止时间的临时代理。

### 4.2 多角色模型

不再把权限只保存在 `profiles.role` 单字段中。新增：

- `roles`：角色定义。
- `permissions`：权限定义。
- `role_permissions`：角色拥有的权限。
- `user_roles`：人员拥有的一个或多个角色。
- `user_regions`：人员允许访问的区域。
- `delegations`：临时代理关系。

首期角色：

| 角色 | 核心权限 |
| --- | --- |
| 老板 | 全局经营汇总、公司利润、全部审计 |
| 管理员 | 账号、角色、区域、功能开关和系统配置 |
| 销售 | 自有客户、线索、商机、报价、订单和跟进 |
| 销售主管 | 下属数据、审批、分配、异常板和订单价差 |
| 财务 | 收付款确认、冲销、内部结算、价差和公司利润 |
| 仓库 | 库存、预留、出入库和缺货异常 |
| 实施 | 待实施订单、安装、培训和实施异常 |
| 运维 | 售后交接、接手确认、续费协作 |

### 4.3 数据权限原则

- 权限必须由数据库执行，前端隐藏按钮不算权限控制。
- RLS 控制可见的数据行。
- 客户电话、采购成本、销售价差和公司利润使用独立敏感表或安全 RPC 返回。
- 普通销售不能批量导出客户。
- 销售主管可以查看下属订单价差，但不能查看公司总利润。
- 公司利润只允许老板和财务查看。
- 所有敏感读取和关键写入均记录审计日志。

## 5. 业务状态机

状态只能通过服务端 RPC 推进，客户端不能直接改最终状态。

### 5.1 线索状态

```text
公海 → 已领取 → 联系中 → 已转商机
                ├→ 培育
                ├→ 回公海
                └→ 无效
```

### 5.2 商机状态

```text
资格确认 → 需求确认 → 演示/方案 → 已报价 → 决策中
                                  ├→ 赢单
                                  ├→ 失败
                                  └→ 培育
```

### 5.3 报价状态

```text
草稿 → 待审批（可选）→ 已发布 → 已接受 → 已冻结
                    ├→ 已过期
                    └→ 已作废
```

### 5.4 订单状态

```text
待定金 → 待内部款 → 待履约 → 履约中 → 待交接 → 已完成
   ├→ 已取消
   └→ 退款处理中 → 已冲销
```

软件履约：

```text
未开始 → 待开通 → 已开通 → 已验证
```

硬件履约：

```text
未开始 → 待备货 → 缺货/已预留 → 已出库 → 已交付
```

### 5.5 实施与售后状态

```text
待排期 → 安装中 → 培训中 → 已完成 → 已交接
```

每次状态变化必须保存操作者、时间、来源状态、目标状态、原因和关联记录。

## 7. 服务端事务与自动化

以下动作必须通过 Supabase RPC 或服务端函数完成：

- 并发领取线索。
- 自动回收和主管重新分配。
- 商机资格重新计算。
- 里程碑日期生成与重算。
- 报价冻结和变更单。
- 财务确认付款、冲销和价差。
- 内部订单创建。
- 库存预留、扣减和释放。
- 订单阶段推进。
- 实施完成和售后交接。
- 续费节点生成。

定时任务：

- 每 15 分钟：执行到期提醒、逾期和回收判断。
- 每天 09:30：发送个人企业微信摘要。
- 每天 10:00：发送次日预约提醒。
- 每 15 分钟：检查预约前 2 小时提醒。
- 失败通知使用递增间隔重试，最多 5 次；仍失败进入后台异常板。

所有自然日按北京时间计算，数据库中时间统一保存为 `timestamptz`，业务计算显式使用 `Asia/Shanghai`。

## 8. 前端信息架构

### 8.1 手机端一级导航

1. 今日
2. 线索清洗
3. 客户
4. 订单
5. 我的

“今日”按以下优先级生成行动队列：

1. 已逾期预约和关键节点。
2. 临近预约。
3. 今日必须跟进。
4. 新线索首次联系。
5. 即将回收。
6. 续费节点。
7. 交付和缺货异常。
8. 普通待办。

每项行动必须直接提供下一步入口，不只展示提醒。

### 8.2 电脑端

- 销售工作台。
- 客户和商机管理。
- 报价与订单管理。
- 主管异常板和会议板。
- 财务确认与对账。
- 库存与履约。
- 实施与售后。
- 产品套餐后台。
- 人员权限、区域和规则设置。
- 审计和导入报告。

### 8.3 功能开关

- 3.0 新页面先使用隐藏路由。
- 功能开关同时控制前端入口和服务端权限。
- 试运行人员加入白名单后才能使用。
- 未开放区域继续使用 2.0，不受影响。

## 9. 安全基线

安全阶段完成前禁止导入真实客户信息。

必须完成：

- 备份现有数据库结构、数据和 Storage 清单。
- 关闭公开注册，验证管理员邀请流程。
- 重构多角色权限并完成权限矩阵测试。
- 检查所有表已启用 RLS，默认拒绝未授权访问。
- 业务表关键写入改为 RPC，撤销客户端直接写权限。
- 客户电话、采购成本、价差、利润与普通字段隔离。
- 私密文件使用私有 Bucket 和短期签名地址。
- 清理浏览器中的敏感业务缓存；Zustand 只保留 UI 状态和可安全恢复的短期缓存。
- 企业微信 Webhook、Supabase Service Role 等密钥只保存在服务端环境变量。
- 审计日志只追加，不允许普通用户修改或删除。
- 定义备份恢复演练和迁移回滚方法。
- 公开仓库执行密钥扫描，提交前和 CI 中都要检查。

## 10. 实施阶段与开工门槛

### 阶段 0：基线与安全底座

交付：

- 当前系统备份和数据库基线。
- 迁移编号规范、回滚规范和测试环境。
- 多角色、区域、临时代理和账号停用。
- 敏感字段隔离、RLS、RPC、审计框架。
- 功能开关和试运行白名单。
- 库存与财务事务风险修复。

验收门槛：

- 跨区、跨人员、跨角色访问测试全部通过。
- 未授权客户端不能直接伪造关键状态。
- 恢复演练成功。
- 仓库和构建产物无服务端密钥。

### 阶段 1：最小销售闭环

交付：

- 品牌、门店、联系人、线索、公海、商机、跟进。
- A/B/C/D 和有效商机判断。
- 手机端“今日”和关键行动入口。
- 产品目录、套餐、报价版本、定金确认。
- 最小订单创建。

试运行：

- 老板、1 名销售、1 名后台人员跑一条模拟订单。
- 通过后再跑一条真实但低风险订单。

### 阶段 2：资金、订单与交付

交付：

- 客户付款、销售代收、汇款、内部采购款、价差和冲销。
- 软件/硬件履约状态。
- 库存预留、出库、缺货异常。
- 安装、培训和售后交接。
- 连锁主订单和多门店交付。

### 阶段 3：管理自动化

交付：

- 自动回收、培育和时间轴重算。
- 主管异常板、销售会议板。
- 企业微信摘要、预约提醒和失败重试。
- 续费 60/30/15 节点。

### 阶段 4：经营数据

交付：

- 季度目标和月度观察。
- 预计积分与官方确认积分。
- 新签 GMV、续费 GMV。
- 官方对账。
- 预测利润和实际利润分离。

### 阶段 5：旧数据导入与逐区开放

交付：

- Excel 模板、预检、去重和试导入。
- 导入结果报告和错误修正表。
- 首批导入 100–500 家客户。
- 按区域逐步开放，保留回退开关。

## 11. 自动化测试与验收清单

### 11.1 权限

- 跨区客户不可见。
- 普通销售不可查看他人客户电话和跟进详情。
- 普通销售不可批量导出客户。
- 销售主管只能查看授权下属的价差。
- 老板和财务以外人员不可查看公司利润。
- 停用账号不能登录或调用 RPC。

### 11.2 线索和商机

- 两人同时领取同一线索只有一人成功。
- 24/48 小时规则正确。
- 15 天回收不被未接电话刷新。
- 三个不同日期未接电话进入培育。
- 有效跟进缺少事实、下一步或时间时不能提交。
- 暂停回收结束后按剩余时间继续。
- A 类未演示不能确认定金。

### 11.3 日期

- 新店 A/B/C 节点计算正确。
- 竞品老店 60/30/15 节点正确。
- 日期变化只重算未完成节点。
- 原日期、修改人和原因可追溯。
- 北京时间跨日边界测试通过。

### 11.4 报价和资金

- 报价版本不可覆盖历史。
- 定金冻结正确目录、成本、积分和 GMV 口径。
- 未结内部款不能开始履约。
- 财务确认定金后才记成交。
- 退款只能冲销，不能覆盖原记录。
- 预计价差和最终价差计算正确。

### 11.5 库存和交付

- 并发出库不产生负库存。
- 库存不足生成缺货异常。
- 取消订单正确释放预留。
- 软件和硬件状态互不覆盖。
- 安装、培训完成后实施可结束。
- 售后交接必须由运维确认。

### 11.6 通知和移动端

- 企业微信失败能够重试并进入异常板。
- 09:30 摘要、次日 10:00 和提前 2 小时提醒正确。
- 手机端关键流程单手可完成。
- 弱网和重复点击不产生重复订单、付款或跟进。

### 11.7 旧数据

- 导入前预检和去重。
- 试导入可撤销。
- 导入数量、失败数量和原因有报告。
- 旧系统汇总与新系统导入结果可对账。

## 12. 发布与回滚

- 每个阶段使用独立 Git 分支开发。
- 每个可验收小功能完成后执行构建、测试和权限检查。
- 通过后提交并推送 GitHub，使用清晰的中文提交说明。
- 数据库迁移只向前追加，不直接修改已经执行的迁移文件。
- 新功能先对白名单开放，再逐区开放。
- 前端异常时可关闭功能开关退回 2.0 入口。
- 数据库迁移必须在执行前完成备份，并准备对应回滚脚本或补偿迁移。
- GitHub Pages 发布后验证实际页面加载的是最新构建文件。

## 13. 团队需要提供的资料

对应阶段开始前，团队提供：

1. 人员名单、账号邮箱、角色、主管关系和区域权限。
2. 区县标准名单和销售可领取区域。
3. 产品、硬件、积分、套餐、加购项和采购成本 Excel。
4. 当前收款、代收、内部采购款、价差和退款的财务口径。
5. 标准销售交接清单和售后群建立要求。
6. 企业微信机器人或应用配置，由管理员在安全环境中录入。
7. 旧客户导入 Excel：品牌、门店、联系人、当前产品、到期日、现负责人。

资料未提供时使用测试数据开发，不自行猜测真实价格、权限或财务金额。

## 14. 开工执行规则

从本方案批准后开始：

1. 先执行阶段 0，不先画大量业务页面。
2. 每次只开发一个可验证的纵向小闭环。
3. 每个阶段完成后由指定试运行人员真实操作验收。
4. 验收不通过不进入下一阶段。
5. 新需求先判断是否属于 3.0；超出边界的进入 3.x 待办，不插入当前阶段。
6. 所有关键规则变更必须修改版本并保留历史，不直接覆盖生产口径。
7. 开发完成后自动测试、提交、推送并发布；遇到账号授权、密钥录入和真实业务判断时暂停，由团队配合确认。

## 15. 最终完成定义

3.0 首期只有同时满足以下条件才算完成：

- 一个真实订单从线索到售后交接完整跑通。
- 一个续费订单完整跑通。
- 权限、并发、回收、日期、资金、库存、退款、通知和移动端验收全部通过。
- 老板、销售、财务/后台、仓库/实施至少各一人完成试运行。
- 首批旧客户导入并完成对账。
- 生产环境连续稳定运行一段试运行周期，无未解决的高风险数据问题。
