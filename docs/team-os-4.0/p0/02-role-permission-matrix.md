> 状态：业务规则、兼容映射、两名成员主岗位决定、P1 物理接口和字段白名单均已冻结；两项人工岗位已在独立恢复项目应用，完整 API 越权矩阵尚未执行。

> 转换登记册：`role-migration-register.md`；机器合同：`scripts/p0/role-migration-contract.json`。

## 身份组合

- 主岗位固定为：销售、实施、运维、财务、管理员。
- 每人恰好一个主岗位；附加业务视图不改变默认首页。
- 仓库/采购是附加职能，不是第六主岗位。
- 主管是叠加职能；主管体系默认关闭。
- 主管体系关闭时，审批、分配、异常和团队目标全部回落管理员。

## Q01 必须覆盖的组合

1. 五个主岗位分别测试，无任何附加职能。
2. 销售/实施/运维分别叠加仓库职能。
3. 每个主岗位分别叠加主管职能，但主管体系关闭。
4. 每个主岗位分别叠加主管职能，主管体系开启且有授权范围。
5. 主管找不到、离职或停用时，责任自动回落管理员。
6. 每种身份直接调用 API，验证看不到无权客户、电话、成本、公司利润和他人收入。
7. 管理员关闭主管体系后，能完成全部审批、分配和异常动作。
8. 批量转移保留原负责人、新负责人、操作人、时间和原因。

## 已冻结的人工岗位决定

- 现网 7 名启用成员中已有 5 名具备唯一主岗位；另外 2 名由项目所有者确认分配为销售 1、管理员 1。
- 生产只读复核确认两人均唯一匹配、启用且目标岗位代码存在；生产岗位写入为 0。
- 姓名和用户 ID 不写入仓库，精确映射只进入仓库外加密恢复制品；先在独立测试项目应用。

## 已冻结的接口补充

- 旧 `owner` 只作为管理员读取兼容别名，不再新增分配；旧 `captain` 只映射主管附加职能且必须另有已确认主岗位；旧 `member` 不参与 4.0 新写授权。
- 权限、会话上下文、导航和责任路由的六个物理 RPC 名已锁定在 `scripts/p0/p1-interface-freeze.json`。
- 主管汇总只允许任务/线索/商机/报价/订单/履约阻塞/续费和付款就绪数量，不允许电话、微信、邮箱、客户收款金额、公司成本/利润及任何他人个人收入。
- 匿名加五主岗位的六类基础身份、16 个附加职能组合和 12 个直接 API 攻击用例编号已冻结；运行结果仍属于 G1 验收，不在 P0 静态阶段伪造。
