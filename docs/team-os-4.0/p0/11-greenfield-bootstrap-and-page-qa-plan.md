# P0-11 全新 4.0 初始化封口与页面基础层验收计划

> 当前边界：先冻结可自动验收条件；未出现初始化实现和六个真实账号运行证据前，G1=false、进度保持 25%。

## 初始化数据库验收

1. 初始化只允许部署端短时高权调用；`anon`、`authenticated` 均没有执行权。
2. 公司、五个主岗位、两个附加职能、首位管理员资料、运行状态和审计在一个事务内完成；任一步注入失败后相关行数全部保持调用前值。
3. 首次成功只产生一个公司、一个首位管理员关系和一组固定字典；不得使用历史固定公司、账号或项目值。
4. 初始化成功同时写入 `initialization_sealed=true` 并永久关闭初始化入口。第二次调用必须明确拒绝，且公司、用户、岗位、职能、运行状态和审计数量及内容指纹均不变化。
5. 封口后继续保持迁移模式：业务写、Cron/工作器和外发副作用均关闭。初始化审计不可更新或删除，且不记录密码、JWT、API key 或数据库凭据。

永久封口撤销的是数据库中 public/private 两个初始化函数的执行权，并清除本地进程中的临时凭据引用。Supabase 项目级 `service_role` key 不能由本工具在服务端单独失效；正式运行后仍须由平台能力轮换/撤销该短时凭据，不能把本地环境变量删除冒充服务端吊销。

数据库自动化分两层：静态门禁核对函数签名、事务/锁、封口、REVOKE/GRANT 和禁止旧根；临时本地 Postgres 单元测试用回滚夹具核对成功、失败注入、二次拒绝和权限攻击。远端只在独立 4.0 项目正式候选执行一次。

## 页面基础层验收

真实页面验收固定覆盖：销售、实施、运维、财务、管理员和第六个拒绝/附加职能身份。每个身份使用真实 Auth 会话，不允许通过 URL 参数或本地模拟角色冒充。

- 五个主岗位固定路由为 `/workspace/sales`、`/workspace/implementation`、`/workspace/operations`、`/workspace/finance`、`/workspace/admin`。
- 页面根提供稳定的岗位测试标识；环境状态提供稳定测试标识。加载成功、无权限、停用账号和未初始化必须能被自动化明确区分。
- 页面只能使用 `apps/team-os-4`、`packages/team-os-4-domain` 和 4.0 Supabase 客户端；禁止引用仓库根 `src`、旧 RPC、旧路由壳和旧迁移链。
- 页面点击验收与直接 API 攻击分别出证据；隐藏菜单不能代替数据库拒绝。

## 30% 门槛

本计划、静态门禁、构建或模拟测试通过均不提高进度。只有独立 4.0 Supabase 持久安装、初始化成功并永久封口、安全复核、六身份真实页面和直接 API 攻击全部通过，才能从 25% 报告到 30%。
