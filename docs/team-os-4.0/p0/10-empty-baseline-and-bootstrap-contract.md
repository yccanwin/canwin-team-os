# P0-10 全新 4.0 空白基线、初始化与搬家隔离合同

> 状态：2026-07-22 路线修订后重新冻结；旧“3.0 原地升级链”作废。尚未执行全新 4.0 空库安装、初始化封口或完整搬家演练。

## 三份明确分离的产物

### 3.0 只读归档

- 现有 3.0 源码、69 个历史迁移、数据库专项备份、Auth、Storage、Functions、Cron 和配置清单只作为恢复、审计和搬家来源保留。
- 3.0 不再接受业务写入；旧前端、定时任务、Webhook、写 RPC、DML 权限和在线高权凭据必须关闭。
- 4.0 安装器、构建和运行时不得执行或引用 3.0 历史迁移、页面、RPC 和数据访问层。

### 4.0 空白基线链

- 从全新 4.0 领域模型生成并审查，不执行 3.0 历史迁移，也不包含历史公司、用户、客户、产品或演示数据的旧种子。
- 只安装结构、必要扩展、基础岗位/权限/状态字典和空产品模板。
- 不包含 `CANWIN_TEAM`、“翻身小队”、历史管理员邮箱、历史项目 ref、真实 URL 或任何访问密钥。
- 演示数据必须是独立可选包，默认不安装。
- 后续迁移只服务 4.0 自身版本升级，禁止为了兼容 3.0 物理结构引入旧表或旧 RPC。

### 3.0→4.0 一次性搬家器

- 位于独立工具根，不进入 4.0 日常运行路径；只读取签名的 3.0 最终快照并写入全新 4.0 目标。
- 固化搬入、合并、归档、丢弃、拒绝和自动新增规则，并永久保存“3.0 来源主键 → 4.0 主键”映射。
- 同一套程序在 G6 后对全新测试项目完整演练一次，在 G7 对全新正式项目执行正式搬家 1/1；真实历史数据不按模块分批导入。
- 正式成功后撤销目标写凭据并封存工具；失败目标不得清洗、复用或原地重试。

4.0 空白安装和“空白安装后执行搬家”的数据库结构必须一致，差异只能来自经过签名映射导入的数据。结构一致性比较至少覆盖：schema、对象名、列、类型、默认值、约束、索引、RLS、策略、视图选项、函数签名/权限、触发器和 Storage 策略。

## 3.0 物理只读与离线来源合同

- 只读不能只依赖员工通知或关闭旧页面。冻结时必须关闭 Auth 注册和刷新、REST/RPC 写入口、Storage 上传、Functions、Cron、Webhook、邮件、通知及所有自动化写任务，撤销普通角色 DML、写 RPC 和在线高权凭据。
- 冻结后使用匿名、普通用户、历史未过期会话和直接 API 分别攻击数据库、Auth 与 Storage；所有写入必须失败。冻结前后重新核对表行数、金额、库存、Auth、Storage 对象和内容指纹。
- 快照导出器只使用数据库级专用只读账号。正式搬家器只读取已离线、加密、签名并校验的快照包，不持有 3.0 URL、JWT、数据库密码或 service-role；3.0 网络可用性不得成为导入依赖。
- 任何来源指纹变化都会使当前快照、搬家包和候选目标失效；必须重新冻结、导出和签名。

## Auth 搬家合同

- 迁移 `auth.users`、`auth.identities` 及经审查的密码哈希字段，优先保留旧 UID；无法保留时必须进入不可变 UID 映射并同步所有业务外键。禁止依赖 profile 自动创建触发器产生重复档案。
- 角色和授权只写入服务端控制的 `app_metadata` 与 4.0 权限表，不从用户可修改 metadata 导入权限。
- 不迁移 sessions、refresh tokens、验证码、恢复令牌或旧项目 JWT secret；4.0 使用独立 JWT secret，所有人员在切换后重新登录。MFA 默认不复制，确需保留时须另列兼容验证和账号恢复方案。
- 对账覆盖用户、身份、禁用状态、邮箱/手机确认状态、UID 映射、员工档案和岗位关系；不得在日志、台账或仓库输出密码哈希和令牌。

## Storage 文件搬家合同

- 数据库中的 `storage.objects` 只算元数据，文件字节必须独立导出、加密、签名和恢复。
- 每个对象冻结 bucket、path、size、MIME、SHA-256、owner、来源业务记录和目标路径。Logo/小程序码两类允许图片进入 4.0，其余历史附件只进加密归档，除非业务清单另行批准。
- 目标禁止覆盖和 upsert；路径冲突、字节哈希不一致或孤儿元数据均立即失败。对象字节与 `storage.objects` 元数据分别全量对账。

## 不可变搬家台账

- 每条来源记录必须且只能归入搬入、合并、归档、丢弃或拒绝之一。
- 台账至少记录 `source_table/source_id → target_entity/target_id`、处置分类、合并或拒绝原因、映射版本、来源快照 SHA-256、代码提交、目标项目和执行批次，并以唯一约束防止一条来源静默映射到多个目标。
- 禁止 `ON CONFLICT DO NOTHING`、无记录跳过、无台账重新生成 ID 和用总行数相等掩盖逐类遗漏。

## 一次性初始化输入

| 输入 | 约束 |
| --- | --- |
| 公司名称 | 必填；一套部署只初始化一家公司 |
| 公司稳定标识 | 由初始化器生成或显式提供，不允许使用历史固定值 |
| 首位管理员邮箱 | 必填；只用于创建当前部署的管理员 |
| 管理员显示名称 | 必填；不得从历史种子带入 |
| Supabase 项目标识 | 必须与当前目标环境合同一致 |
| 访问地址 | 仅登记当前部署地址，不写入源码默认值 |
| 时区 | 固定 `Asia/Shanghai` |
| 主管体系 | 默认关闭 |
| 功能开关 | 仅启用已通过当前阶段门禁的能力 |

密码、service role、数据库密码和临时管理凭证不是初始化文件字段，只能从部署者本机或受控服务端的临时安全环境读取。

## 初始化事务

一次初始化必须作为受控、幂等、可审计流程完成：

1. 校验目标项目 ref、空白状态和迁移版本。
2. 创建唯一公司记录。
3. 创建五个主岗位字典，并将仓库、主管定义为附加职能而非主岗位。
4. 创建首位管理员 Auth 用户和业务 profile/岗位关系。
5. 写入基础状态字典、默认关闭的主管开关和阶段功能开关。
6. 写入初始化审计，记录版本、执行人、时间、目标 ref 和结果，不记录凭证值。
7. 提交成功后撤销临时管理凭证并永久关闭初始化入口；Cron/工作器继续保持关闭，直到完整搬家、全量对账和五个真实岗位账号验收全部通过。

任一步失败必须整体回滚；不得留下“公司已建但管理员未建”或“管理员已建但入口未封”的半初始化状态。

## 初始化封口

成功后必须同时证明：

- 第二次调用返回“已初始化”且不产生新公司、用户、岗位或审计重复记录。
- 浏览器和匿名 API 不存在创建首位管理员的入口。
- 临时管理凭证已撤销或失效，仓库和日志中没有凭证值。
- 初始化接口的执行权限已撤销，或部署期工具已从运行环境移除。
- 初始化封口后进入“迁移模式”：Cron、工作器、通知、邮件、Webhook、外发 Edge Function 和会产生业务副作用的自动化触发器全部关闭，只保留完整性约束。
- 完整搬家、全量对账和五个真实岗位账号验收通过后，才按批准清单启用 Cron/工作器，并使用 `Asia/Shanghai`。

## 空库验收

| 验收项 | 必须证据 |
| --- | --- |
| 安装来源 | Git 完整 SHA、空白基线版本和依赖锁文件 |
| 数据纯净 | 员工、客户、品牌、门店、联系人、订单、财务、日志、案例、图片和 Auth 历史数量为 0 |
| 基础字典 | 五主岗位、附加职能、状态字典和空产品模板符合冻结合同 |
| 首位管理员 | 只创建 1 个当前部署管理员，能够登录并完成管理员动作 |
| 初始化封口 | 第二次初始化失败且无副作用；匿名/普通用户无法调用 |
| 环境隔离 | 目标 ref 是独立测试项目，不等于生产 ref |
| 结构一致 | 与同版本 4.0 空白基线的规范化 catalog 一致；搬家只产生获准数据差异 |
| 安全顾问 | 新增对象无 ERROR；显式 GRANT、RLS、策略和函数执行权通过 |
| 调度 | Cron/工作器时区、游标、补跑和禁重结果通过 |

## 施工边界

- P0 只冻结输入、事务、封口和验收合同，不在 3.0 项目创建管理员、修改结构或重跑历史初始化。
- 空白基线只能在全新独立 4.0 测试项目生成和验证，不从 3.0 导出后“清空几张表”冒充。
- `1/1` 表示每个候选目标项目只允许执行一次正式安装或搬家。第一次失败后立即停止，保留项目、日志、版本、来源快照、搬家包和 catalog 快照；下一候选必须重新核验来源冻结并生成新签名快照、新搬家包、新目标项目和新短时凭据，失败候选的任何一项均不得复用。
