## 额外路由

- `/`：当前是 Dashboard 别名；4.0 应与 `/dashboard` 使用相同岗位工作台规则。
- `/notifications-v3`：方案未单列，但 4.0 要求消息固定在顶栏；需保留兼容地址并提供顶栏入口。
- `/access-v3`：当前已显式重定向到 `/settings-v3/access`，可作为兼容路由实现样板。

## 导航与岗位壳层事实

- 桌面导航当前是所有用户共用的静态常量，不按主岗位、仓库职能或主管开关生成。
- 当前角色仍以 `admin/captain/finance/warehouse/member` 为主，与五主岗位模型不一致。
- 当前移动底栏不是方案固定的“工作台、推进、日历、岗位业务、我的”。
- 账户菜单没有附加工作视图切换，消息也未固定在顶栏。
- `/sales-v3` 的内部导航与全局导航并存，桌面存在双侧导航。

## 任务来源事实

- Dashboard、Work、Tasks、Calendar 对 `tasks` store 有部分同源基础。
- Calendar 还拼接独立日程、团队目标和个人目标。
- 销售今日行动另走独立 RPC。
- 尚无覆盖销售、履约、财务、库存业务事件的统一 `WorkItem` 来源。

## P1/P2 前必须冻结

1. `AppContext`：公司、成员、主岗位、附加职能、主管开关、权限、工作视图。
2. `NavigationManifest`：routeId、顺序、分组、canonicalPath、visible、enabled、readOnly。
3. `WorkItem`：来源、负责人、期限、优先级、状态、下一步、阻塞、唯一生成键和业务动作。
4. `CalendarProjection`：工作项只读投影 + 独立 ScheduleEvent，禁止复制工作项。
5. 4.8 全量 canonical/redirect/hidden/readOnly 映射及旧收藏深链测试。
