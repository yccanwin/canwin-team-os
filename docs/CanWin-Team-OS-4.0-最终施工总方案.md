# CanWin Team OS 4.0 最终施工总方案

> 文档状态：施工基线版
> 日期：2026-07-18
> 适用项目：CanWin Team OS 4.0
> 技术底座：React + TypeScript + Vite + Supabase
> 施工组织：1 名监理统筹，3 个施工团队并行
> 发布原则：增量建设、兼容迁移、分角色验收、失败即停、可回退

## 1. 最终建设结论

4.0 不是在现有页面上继续堆功能，也不是建设多公司 SaaS。4.0 的最终定义是：

> 面向客如云代理商团队的单公司经营操作系统。每套源码只服务一家公司；其他代理商获得空白源码后，独立搭建、独立数据库、独立账号、独立数据。

系统底层围绕“人、货、场、钱”四类业务对象建立统一规则；普通员工不直接看到四大中心，而是看到符合自己岗位的工作台、推进任务和业务入口。管理员通过管理页面查看四类底层经营信息。

4.0 成功的标准不是页面数量，而是以下主链能够由真实岗位完整跑通：

```text
线索 → 客户/门店 → 商机 → 报价 → 客户付款 → 正式订单
     → 内部款结清 → 软件/硬件/服务履约 → 实施/运维交接
     → 收益结算 → 案例候选 → 续费
```

## 2. 不可变更的施工边界

以下规则作为 4.0 施工基线，后续团队不得自行改变：

1. 单公司、单数据库、单独部署，不建设多公司切换、公共租户后台和统一计费平台。
2. 保留数据中的公司边界字段，但前端不允许写死 `CANWIN_TEAM` 或“翻身小队”；每次部署初始化一个公司。
3. 普通员工不显示“人、货、场、钱”四大中心；四类信息只通过岗位工作台和管理员管理页面呈现。
4. 主管体系默认关闭，由管理员承担全部主管审批、分配和异常处理；团队扩大后可全局开启主管体系。
5. 一个人有一个主岗位，可拥有多个附加角色或技能；主岗位决定默认首页和侧边栏，附加角色决定可处理的额外任务。
6. 全站取消普通图片和附件上传。案例馆只允许品牌/门店 Logo 和小程序展示码各一张。
7. 公司自身 Logo 作为部署配置资产，不提供普通员工上传入口。
8. 不删除、不改名、不重写现有 69 个迁移文件；所有 4.0 数据库变化只允许新增兼容迁移。
9. 报价不占库存。正式订单生成并满足内部付款条件后，才能预留和出库。
10. 财务确认、退款、冲销、内部款、销售利润和劳动收益必须保留不可篡改的事件记录。
11. 当前 3.0 在施工期继续运行。4.0 使用隐藏入口和功能开关建设，未验收角色不得切换。
12. 第一次正式迁移、部署、恢复或发布失败后立即停工，保留证据，不自动重试。

## 3. 使用者与岗位模型

### 3.1 固定岗位

| 岗位 | 默认职责 | 默认可见范围 |
| --- | --- | --- |
| 销售 | 线索、客户、商机、报价、订单、回款协作、续费 | 自己负责及被授权区域的数据 |
| 实施 | 排期、安装、培训、验收、交接 | 分配给自己的订单和门店 |
| 运维 | 售后、客户维护、异常、续费协作、服务交付 | 分配给自己的客户和服务任务 |
| 财务 | 收付款确认、内部应付、冲销、利润和收益结算 | 全部财务记录及必要订单摘要 |
| 管理员 | 所有主管职责、人员权限、经营配置、异常处理 | 全局数据和审计记录 |

4.0 首版只设以上五个主岗位。仓库/采购是附加职能能力，默认由管理员兼任，也可授予实施人员，不建立第六个独立主岗位和独立首页。主管也是叠加在主岗位上的治理角色，例如“销售主岗位 + 销售主管”，不建立脱离业务岗位的通用主管首页。

一个人只能有一个主岗位，可以拥有其他业务工作视图、仓库职能和主管职能。主岗位决定默认首页；普通附加业务角色通过右上角账户菜单切换工作视图；仓库和主管职能在当前主岗位侧边栏增加对应的“仓库处理”或“团队审批”分组，不混入其他无关页面。

普通员工没有全局财务明细权限，财务和管理员可以查看全局资金账。

### 3.2 主管开关

系统设置增加“启用主管体系”总开关：

- 默认关闭：所有需要主管确认的地方显示“待管理员确认”，审批、分配、异常和团队目标均由管理员处理。
- 开启后：管理员为主管分配区域、人员和业务范围；匹配范围内的事项交给主管处理。
- 找不到主管、主管离职或主管停用时，事项自动回退管理员。
- 管理员始终保留查看、接管和改派权限。
- 开关变化只改变责任路由，不改变历史订单、审批人和操作记录。

### 3.3 权限原则

- 前端隐藏按钮不等于权限；所有敏感读写必须由数据库授权、RLS 和安全函数共同执行。
- 员工只能看到完成本岗位任务所需的信息。
- 客户电话、内部价、采购成本、销售利润、公司利润、劳动收益分别授权，不能通过一张宽表全部返回。
- 离职先停用账号，再转移客户、商机、订单和任务；历史记录保留原负责人。
- 所有批量转移和代办必须记录原负责人、新负责人、操作人、时间和原因。

## 4. 前台信息架构

### 4.1 所有人共有

1. 我的工作台：只显示今天需要处理的事项和岗位数据。
2. 推进中心：统一任务队列，可筛选全部、今天、逾期、等待他人、已完成。
3. 日历：显示同一批推进任务的时间视图，不另建一套日程数据。
4. 消息：显示审批结果、异常、到期和任务变更。
5. 个人主页：显示本人岗位、技能、任务完成情况和本人可见收益。

桌面端菜单顺序固定为：我的工作台、推进中心、日历、当前岗位业务；有仓库职能时追加“仓库处理”，有主管职能且主管体系开启时追加“团队审批”。消息固定在顶栏，个人主页和工作视图切换固定在右上角账户菜单。

工作台、推进中心和日历必须共用一套任务来源。业务事件可以生成或关闭任务，禁止三个页面各自维护重复待办。

### 4.2 销售侧边栏

- 线索与公海
- 客户与门店
- 商机与跟进
- 报价与订单
- 回款与内部款状态
- 续费客户
- 我的业绩、积分和利润

### 4.3 实施侧边栏

- 待排期
- 待安装
- 待培训
- 待验收
- 待交接
- 实施异常
- 我的服务收益

### 4.4 运维侧边栏

- 服务队列
- 客户维护
- 售后异常
- 续费协作
- 案例素材待补充
- 我的服务收益

### 4.5 财务侧边栏

- 待确认收款
- 销售代收与内部应付
- 退款与冲销
- 销售利润结算
- 劳动收益结算
- 公司成本与现金支撑

### 4.6 管理员侧边栏

- 经营驾驶舱
- 审批与分配
- 组织、岗位与权限（人）
- 商品、价格、仓库与服务（货）
- 客户、品牌与门店（场）
- 财务、成本与结算（钱）
- 案例馆与网站发布
- 系统设置

管理员页面可以体现四类底层信息，但侧边栏不直接使用抽象的“人中心、货中心、场中心、钱中心”命名。

以上 4.2–4.6 均为“当前岗位业务”分组，不再各自增加第二个工作台入口。所有人只保留一个“我的工作台”，系统根据当前工作视图显示销售、实施、运维、财务或管理员的今日内容；管理员的“经营驾驶舱”是经营管理分析页，不是第二个个人工作台。

### 4.7 移动端边界

移动端底栏固定为：工作台、推进、日历、岗位业务、我的。岗位业务分别指销售客户、实施任务、运维服务、财务收款和管理员审批；消息仍在顶栏。移动端只承担高频日常动作：查看今日任务、记录跟进、联系客户、处理实施/运维步骤、查看订单状态和完成个人任务。商品目录维护、财务总账、权限配置、批量导入和复杂报表仅保证桌面端完整使用。

## 5. 统一推进引擎

推进中心是 4.0 的日常工作中枢。每条工作项至少包含：来源业务、来源记录、岗位类型、负责人、截止时间、优先级、状态、下一步、阻塞原因和生成规则。

工作项分为两类：

- 提醒/手工任务：不直接改变核心业务状态，可以使用通用“完成”。
- 业务动作任务：例如确认收款、出库、验收、冲销，不提供通用“完成”；用户必须执行对应业务动作，服务器事务成功后由业务事件自动关闭任务。

推进中心不得把“点击完成”当成绕过报价、财务、库存或履约状态机的入口。

统一状态：

```text
待处理 → 处理中 → 等待他人 → 已完成
   └────────────→ 已取消
```

排序规则从高到低为：

1. 已逾期且阻塞成交、履约或收款；
2. 今天必须完成；
3. 临近客户预约、开业或到期时间；
4. 新线索首次联系；
5. 即将回收的线索；
6. 续费 60/30/15 天节点；
7. 普通任务。

系统自动生成业务任务，员工可以创建个人任务，但个人任务不得改变订单、财务和库存状态。所有列表必须支持关键词搜索、组合筛选、保存视图和清除筛选；人员、客户、门店和商机选择器统一使用可搜索选择框。

日历同时展示两类内容：统一工作项的计划/截止时间，以及会议、拜访、休息日和个人行程等日程事件。工作项在日历中只是一种视图，不复制成日程数据；手工日程可以关联客户或工作项，但不能代替业务任务。

## 6. “场”：客户、品牌、门店和生命周期

### 6.1 数据关系

```text
结算客户（签约/欠款主体）
  └─ 品牌
      ├─ 门店 A
      ├─ 门店 B
      └─ 门店 C
```

每张报价和订单必须归属一个结算客户，付款、欠款、退款和开票口径均归该结算客户。一个订单只能包含同一结算客户下的门店；不同加盟商或不同结算主体必须拆分订单。

联系人通过“联系人—门店关系”关联一个或多个门店，并且每个联系人最多有一个主要门店关系；电话等敏感信息继续独立授权。品牌联系人也通过关系记录表达，不能继续依赖联系人表中的单个 `store_id` 作为唯一关系。

订单可以覆盖同一结算客户下的多个直营门店。每条报价/订单商品行必须分配到一个或多个门店，门店分配数量之和必须等于订单行数量；软件、硬件和服务均按“门店 × 订单行”形成履约单元。客户付款和退款保持订单级，交付、验收、异常和续费保持门店履约单元级。

续费事实的最小粒度固定为“门店 × 商品订阅”，每个订阅独立保存开通日、到期日和原订单。品牌层只能聚合查看，不能用一个品牌到期日覆盖所有门店和产品。

### 6.2 线索创建

线索创建只要求最少信息：称呼或标题、联系电话、来源、区域、新店/竞品老店/已有客户/未知。地址选填。

- 默认带出创建人的主区域，但允许选择其他启用区域。
- 创建到本人无权限区域时，不自动归本人所有；进入对应区域公海或分配给该区域销售。
- 按电话、门店名称和地址进行重复提醒，不允许静默创建明显重复记录。
- 误建且从未跟进、未关联客户/报价/订单的线索，创建人可在 24 小时内删除。
- 已跟进线索使用“放弃”，回到区域公海。
- 公海 20 天无人领取且没有受保护业务记录时物理删除，只保留不含电话和客户内容的删除审计。
- 已产生报价、订单、收付款或履约记录的客户数据不得物理删除，只能标记失败、停用或归档。

20 天清理的唯一规则：领取动作和清理任务使用同一行锁，领取优先；领取成功立即清空清理时间。存在商机、报价、订单、收付款、履约、订阅或案例任一记录时绝不删除。只有为该线索单独创建且未被其他业务引用的临时联系人/门店、跟进、负责人历史可以随线索清理；电话和跟进正文删除，最小审计只保留随机编号、区域、删除时间、删除原因和任务版本，不保留可识别客户内容。

### 6.3 新店流程

新店进入主动推进：完善必要资料、确认开业日期、资格判断、商机、报价、订单和交付。开业日期变化时，只重新计算未完成节点，已完成节点保留。

### 6.4 竞品老店流程

竞品老店完善基本资料和当前系统到期日后进入沉淀商机池：

- 到期 60 天前：自动回到销售工作台，完成情况摸底；
- 到期 30 天前：完成方案和报价；
- 到期 15 天前：确认决策或记录结构化失败原因；
- 未进入 60 天窗口前，不占用日常活跃商机列表；
- 到期日未知时必须设置下一次核实时间，不能无限沉淀。

### 6.5 快捷推进与跳步

老客户介绍、客户主动购买、已明确需求等场景允许使用“快捷推进”：

- 可以从线索直接创建客户/门店和商机；
- 资料齐全时可以直接进入报价；
- 已谈妥客户可以使用快速建单向导记录必要过程；
- 所有跳过步骤必须记录原因和操作者；
- 客户/门店基本身份、报价快照、付款确认、内部款、库存和履约条件不得跳过。

### 6.6 客户 360 浮层

客户和门店不再依赖多张独立详情页来回跳转。点击客户后打开 360 浮层，集中显示：基本资料、联系人、门店、负责人、最近跟进、商机、报价、订单、收款、履约、续费和案例状态。复杂编辑使用分段抽屉，不使用一次填写全部字段的超长表单。

### 6.7 跨岗位操作矩阵

| 操作 | 销售 | 实施 | 运维 | 财务 | 管理员 |
| --- | --- | --- | --- | --- | --- |
| 提交线索 | 可创建并经营本人/授权线索 | 仅 `leads.submit` 报备 | 仅 `leads.submit` 报备 | 不可 | 可 |
| 编辑客户 360 | 负责范围内商业资料 | 仅交付资料 | 仅服务和到期事实 | 仅付款/结算事实 | 全部 |
| 补充案例文字 | 参与订单可 | 参与交付可 | 参与服务可 | 不可 | 全部 |
| 上传案例两张图片 | 不可 | 不可 | 不可 | 不可 | 可 |
| 修改续费 | 负责商业推进、报价 | 仅补交付事实 | 可补服务事实和到期证据 | 仅收款状态 | 全部 |
| 发布/下架案例 | 不可 | 不可 | 不可 | 不可 | 可 |

实施和运维提交线索时只保留提交人，必须选择区域和销售或进入区域公海，不自动成为负责人；提交人只能查看自己提交线索的处理状态。快速建单向导不能绕过报价，它必须在后台自动生成并冻结一份完整报价快照，再创建正式订单。

## 7. “货”：软件、硬件和服务

### 7.1 统一商品目录

目录统一支持三种商品：软件、硬件、服务。每个商品版本保存：名称、SKU、类型、公司给销售的内部价、建议客户价、最低允许价、公司预计实际成本、积分、是否需要履约、服务岗位/技能、劳动收益规则和有效期。

三种价格必须分别保存、分别冻结，禁止复用同一个字段：

1. 客户成交价：销售给客户的实际价格。
2. 销售内部价：销售应结算给公司的价格，也是销售利润的扣除基数。
3. 公司实际成本：公司为软件、硬件、劳动收益及其他直接交付支付的成本，仅财务和管理员可见。

公司订单毛贡献按“销售内部价减公司实际直接成本”计算；工资、社保和日常成本在管理账中另行扣除，用于经营利润和现金支撑判断。目录中的成本是预计值，订单可以冻结财务确认后的实际成本快照，不能用“销售内部价”冒充采购成本。

目录发布后不可原地修改；新价格通过新目录版本发布，历史报价和订单继续使用冻结快照。

### 7.2 服务商品

服务也是一种货物。例如“小程序设计”：

```text
客户成交价 500
公司给销售内部价 300
销售利润 200
执行人员劳动收益 150
公司服务毛收益 150
```

- 服务可以绑定岗位和技能，不永久绑定某一个人。
- 系统按岗位、技能和当前任务量推荐执行人。
- 主管体系关闭时由管理员确认执行人；开启后由相应主管确认。
- 劳动收益在服务完成并验收后形成应付，在结算日支付。
- 订单冻结客户价、内部价、劳动收益和积分规则，后续改价不影响历史订单。

### 7.3 硬件与仓库

- 报价阶段不检查和占用库存。
- 正式订单冻结硬件 SKU 和数量。
- 满足内部付款条件后才能预留、出库和交付。
- 出库必须扣减同一仓库库存并产生库存流水。
- 禁止负库存、额外 SKU、超数量出库和完成时残留预留。
- 缺货生成异常、预计到货日和处理任务。
- 取消未出库订单时释放预留；已出库只能通过退库/冲销流程处理。

## 8. “钱”：付款、内部款、利润和劳动收益

### 8.1 四类金额

1. 客户成交价：客户最终应付金额。
2. 内部应付：销售取得产品或服务必须支付公司的金额。
3. 销售利润：客户净收款扣除内部应付及销售承担费用后的余额。
4. 劳动收益：实施、运维或其他执行人员完成服务后取得的劳动所得。

### 8.2 收款方式

客户可以付款给公司，也可以付款给销售，同一订单允许混合收款。每笔收款必须记录付款方、收款方、金额、方式、时间、订单和确认状态。

每笔客户款先按实际收款人分别进入“公司持有”或“销售持有”，但订单净收款统一计算。资金顺序冻结为：

```text
确认客户净收款
→ 公司持有款先覆盖内部应付
→ 销售持有款中应上缴部分 + 销售补款覆盖剩余内部应付
→ 经批准的公司补贴覆盖低于内部价的差额
→ 全款且内部应付已覆盖后确认销售利润池
→ 按订单冻结比例分配利润，并按实际持款形成公司应付或销售应缴
```

部分收款即使已经超过内部应付，也只形成“预计利润”，客户全款和内部应付同时满足前不得结算。收款人不等于利润归属人：多人协作时，实际持款与订单利润比例分别记账，持款超过本人应得部分时形成销售应缴，持款不足时形成公司应付。

施工计算口径固定为：

```text
订单净收款 = 已确认客户收款 - 已确认客户退款
公司可用于内部款 = 公司持有净款中分配给本订单的金额
内部款覆盖 = 公司可用于内部款 + 销售已上缴 + 销售补款 + 已批准公司补贴
内部款未结 = max(销售内部价合计 - 内部款覆盖, 0)
销售利润池 = max(min(订单净收款, 客户成交价合计) - 销售内部价合计 - 销售承担费用, 0)
个人应得利润 = 销售利润池 × 订单冻结利润比例
```

销售利润只有在“订单净收款达到客户成交价合计”且“内部款未结为零”时从预计转为可结算。公司持有超过内部款的部分形成公司应付销售款；销售实际持有超过本人应得利润且未上缴的部分形成销售应缴。客户退款先重算订单净收款，再按新利润池追加反向调整；公司补贴只覆盖经批准的内部价缺口，不扩大销售利润池。

销售代收时，未结清内部应付不得启动软件开通、硬件备货和服务履约；销售可以自行补足差额。销售欠内部款时暂停新的利润发放和代收资格，但不停止其正常销售工作。

### 8.3 财务确认

- 所有资金事件先进入“待确认”。
- 只有财务或管理员可以确认、驳回和冲销。
- 已确认记录不能编辑或删除，只能新增反向记录。
- 公司低于内部价成交且经过批准时，销售利润最低为零，内部价缺口由公司补贴事件覆盖，不扣减已确认的执行人员劳动收益。
- 正常履约后的内部价原则上不退；非公司责任的客户退款由销售利润承担。
- 公司责任退款进入异常审批，明确公司和销售承担额。已验收形成的劳动收益不得自动冲减或追回；如存在人员责任，只能另建有审批、原因和上限的责任事件，不得修改原收益流水，也不得自动从已支付劳动收益扣款。

已结算利润后发生退款时，原利润支付记录不修改；系统新增反向调整。销售尚未支付的利润先抵扣，不足部分形成销售债务，并按既定规则暂停新的利润发放和代收资格。公司补贴、销售补款、销售应缴、公司应付和退款调整均为独立流水，不能互相覆盖。

### 8.4 业绩、积分与多人分配

- 业绩和积分只从订单冻结快照自动产生，停止手工加分作为主要流程。
- 财务确认全款后确认业绩和积分；净收款低于订单总额时整单回滚，补足后恢复。
- 新签 GMV 和续费 GMV 分开统计。
- 多销售协作时，利润比例在订单生效前冻结；积分归主销售。
- 员工只看自己的利润或劳动收益，财务和管理员看全部；主管只能看授权团队汇总，不看无关人员个人收入。

### 8.5 公司成本与现金支撑

“现金还能支撑多久”只作为管理口径，不替代法定会计。数据源固定为：资金账户期初余额、已确认现金收支、受限资金、已承诺未支付内部采购/劳动收益、未来工资和社保计划、固定日常成本计划。

```text
可用现金 = 已确认账户余额 - 受限资金
未来30天承诺支出 = 到期采购 + 劳动收益 + 工资社保 + 固定成本 + 其他已批准应付
滚动日均净消耗 = 最近90天经营现金净流出 / 有效经营天数
预计支撑天数 = 可用现金 / 滚动日均净消耗
```

有效数据不足 30 天或滚动日均净消耗不为正时，不显示虚假天数，只显示可用现金和未来 30 天已承诺支出。期初余额和成本计划只能由财务录入、管理员确认，并保留审计。

## 9. 案例馆与网站联动

### 9.1 生成流程

```text
门店履约完成 → 自动生成内部案例候选 → 销售/实施/运维补充文字
→ 管理员审核 → 发布案例馆 → 网站读取已发布案例
```

成交不等于自动公开，管理员审核也不能代替客户展示授权。案例必须保存授权状态、授权来源、允许展示范围、生效时间、失效时间和撤回记录；授权证明使用结构化文字和外部凭证编号，不上传附件。无有效授权的案例只能保留为内部候选。管理员可以编辑、发布、下架和归档。

案例候选按“门店履约”生成，同一多门店订单可以形成多个门店案例；品牌案例页只聚合已授权、已发布的门店案例，不直接按整张订单生成一份混合案例。

### 9.2 案例字段

- 品牌和门店展示名称
- 行业、区域、新店/老店
- 使用的软件、硬件和服务
- 客户原有问题
- 解决方案
- 上线结果
- 服务团队展示信息
- 品牌/门店 Logo
- 小程序展示码
- 发布状态和排序
- 客户展示授权状态、范围和有效期

公开内容不得包含联系人、电话、详细地址、报价、内部价、销售利润、公司利润和内部操作记录。

公司网站通过独立的“已发布案例公开投影/API”读取脱敏字段，匿名访问者永远不能直接查询内部案例表。草稿图片存放在私有案例桶；发布时把经过授权的 Logo 和小程序展示码复制到独立公开案例目录。撤回授权或下架案例时，公开投影立即不可见并删除公开副本，内部候选仍保留私有记录。

### 9.3 图片禁令

- 案例只允许 Logo 1 张和小程序展示码 1 张。
- Logo 压缩后不超过 200KB；展示码不超过 300KB。
- 只允许 PNG、JPG、WebP；禁止 SVG、GIF 和其他文件。
- 前端、Storage 路径、数据库接口三层共同限制。
- 只有管理员可以上传或替换两个案例图片；销售、实施和运维只能补充自己参与案例的结构化文字。
- 现有团队相册、照片上传、成就图片和其他上传入口从 4.0 导航和权限中移除。
- 数据库备份不等于文件备份，两张允许图片必须单独纳入文件备份清单。
- 测试环境从 P0 开始执行图片禁令；生产环境在正式切换前先关闭旧上传入口，再撤销通用 Storage 写策略。施工期不得提前修改生产策略而破坏仍在运行的 3.0。

## 10. 数据与技术施工原则

### 10.1 现状基线

2026-07-18 只读审计显示：现网约 103 张 public 表、162 个 public 函数、229 条策略、248 个索引、46 个触发器；本地已有 69 个迁移文件。现网数据量较小，但数据库安全和性能顾问仍有需要先处理的错误和告警。

当前三个上线前必须清除的安全错误是 `finance_public_summary`、`inventory_public_items`、`assets_public` 三个安全定义者视图。现有高权限函数不能整体删除，必须逐个区分正式业务 RPC、内部触发器/助手和废弃函数，再核验身份检查、岗位范围和执行授权。

### 10.2 数据分类

阶段 0 必须把现有对象逐项分为四类：

| 分类 | 处理方式 |
| --- | --- |
| 保留 | 结构和业务语义符合 4.0，只补权限、索引和接口 |
| 扩展 | 保留旧字段，新加 4.0 字段、事件表或适配视图 |
| 只读 | 旧页面停用但历史数据仍需查询 |
| 淘汰 | 先断开新写入，保留 30 天，只在验证无引用后另立删除方案 |

第一版禁止直接删除表、改表名、改历史迁移和批量重写现有数据。

### 10.3 4.0 数据模块

- 人：账号、主岗位、附加角色、技能、区域、代理关系、收益账户。
- 场：结算客户、品牌、门店、联系人—门店关系、线索、商机、跟进、生命周期、门店商品订阅和续费节点。
- 货：目录版本、软件、硬件、服务、套餐、报价快照、库存和服务分配。
- 钱：客户收款、内部应付、冲销、销售利润、劳动收益、成本和审计。
- 推进：统一工作项、生成规则、状态变化和消息事件。
- 案例：案例候选、发布内容和两个受限图片位。

具体表名在阶段 0 完成对象盘点后冻结，禁止三个团队各自创建含义重复的表。

### 10.4 Supabase 安全要求

- 所有暴露表必须显式 `GRANT`、启用 RLS 并建立最小权限策略，三项在同一迁移中交付。
- 不用用户可修改的 metadata 做授权判断。
- 视图默认使用调用者权限；确需提权的函数放入非暴露 schema，检查当前用户并撤销 PUBLIC 执行权限。
- 前端永远不能包含 service role 或数据库密钥。
- 所有财务、库存、领取、状态推进和收益结算使用幂等事务函数。
- 每个外键、常用筛选、到期排序和任务队列建立必要索引。
- 每次候选迁移必须先通过安全顾问、性能顾问、权限矩阵和空库安装验证。
- 当前发布工作流只有安装、构建和发布，4.0 必须增加统一测试入口、实际测试数量、权限测试、迁移检查和安全门禁；仅“构建成功”不再代表候选可发布。

### 10.5 初始化、调度和漏跑补偿

4.0 新安装不能通过公开前端创建首位管理员。交付包提供一次性可信初始化程序，在部署者本机或受控服务端使用临时管理凭证创建单公司、首位管理员、基础岗位和功能开关；成功后立即撤销临时凭证并永久关闭初始化入口。

公海 20 天清理、老店/续费 60/30/15 天提醒、通知重试和结算到期检查由受控 Cron/工作器执行，统一使用 `Asia/Shanghai` 计算业务日期。每个调度任务必须幂等，保存上次成功游标，并能在停机后补跑遗漏窗口；密钥只保存在托管环境，不进入前端和源码仓库。

## 11. 三个施工团队职责

### 11.1 监理

监理只负责计划、接口冻结、任务分配、代码审查、跨团队整合、阶段验收和发布决策，不代替团队绕过验收。

监理拥有以下否决权：

- 业务规则与本方案不一致；
- 权限只在前端处理；
- 新旧数据无法兼容或回退；
- 真实数据被写入测试环境；
- 没有对应测试和验收证据；
- 团队私自重试正式迁移或发布；
- 图片上传超过允许范围；
- 将单公司系统擅自扩展为多公司平台。

### 11.2 团队一：后端业务与数据

负责数据库对象盘点、业务状态机、权限、RPC、定时规则、索引、审计、财务和库存一致性。团队一不制作最终页面，不直接发布生产。

核心工单：

- B01：103 张表及相关函数、策略、触发器四分类清单。
- B02：单公司初始化和当前用户公司解析，移除业务接口中的写死团队值。
- B03：主岗位、附加角色、技能、区域和主管开关。
- B04：统一工作项和业务事件生成规则。
- B05：新店、竞品老店、续费和沉淀商机状态机。
- B06：线索删除、放弃、公海 20 天清理和删除审计。
- B07：结算客户、品牌、门店、联系人多门店关系和 360 聚合接口。
- B08：软件、硬件、服务统一目录，三种价格及版本快照。
- B09：报价、门店行分配、正式订单、混合收款瀑布、内部应付和多人利润冻结。
- B10：库存预留、出库、退库和缺货异常。
- B11：实施、运维、服务分配、验收和劳动收益。
- B12：收款、代收、补款、退款、冲销和结算。
- B13：业绩、积分、新签/续费自动计算。
- B14：案例候选、发布视图和两个图片位的存储策略。
- B15：安全、性能和幂等性收口。

### 11.3 团队二：前端产品与交互

负责岗位壳层、侧边栏、工作台、推进中心、客户 360、业务操作界面、搜索筛选、响应式和真实错误反馈。团队二不得在前端复制后端业务判断。

核心工单：

- F01：按主岗位和权限生成侧边栏，移除重复双侧导航。
- F02：共同工作台、推进中心、日历和消息壳层。

## 13. 阶段验收门禁

### G0：允许写业务代码

- 数据库和 Storage 完整备份可用。
- 数据库、Auth 身份/岗位关系和 Storage 已在独立项目成功恢复一次；没有恢复证据不得执行第一次生产安全迁移。
- 103 张表四分类完成。
- 结算客户—品牌—门店、联系人多门店、三种价格、混合收款、多门店履约、续费和案例最小粒度已冻结。
- 当前安全顾问 3 个错误已有修复方案并在隔离环境验证。
- 关键外键索引和权限风险已有优先级清单。
- 测试项目与生产项目完全隔离。
- 建立统一自动测试入口，并让持续集成实际执行数据库、权限和业务测试，而不是只执行前端构建。

### G1：岗位和权限通过

- 五个主岗位分别验证，仓库附加职能和主管附加职能分别验证开/关组合。
- 主管关闭时普通界面不出现主管入口，管理员能完成全部主管动作。
- 任一角色通过直接 API 也不能读取越权数据。

### G2：推进中心通过

- 工作台、推进中心和日历展示同一来源工作项。
- 完成任务只产生一次业务推进，多次点击不重复记账或重复建单。
- 逾期、等待他人、续费和普通任务排序正确。

### G3：销售和门店通过

- 新店完整流程、竞品老店沉淀流程、到期日未知流程全部通过。
- 跨区域创建、重复提醒、放弃、公海和 20 天清理通过。
- 客户 360、快捷推进、搜索筛选和保存视图可用。

### G4：货和履约通过

- 报价绝不占库存。
- 正式订单冻结软硬件服务和金额快照。
- 未结内部款无法预留和出库。
- 缺货、退库、取消、服务分配、实施和运维交接可闭环。

### G5：钱和收益通过

- 公司收款、销售代收、混合收款、补款、退款、冲销全部对账一致。
- 销售利润和劳动收益不能互相覆盖。
- 全款确认、欠款回滚、补足恢复和多人分配符合规则。
- 普通员工无法读取公司利润和他人收入。

### G6：岗位成品通过

- 五个岗位分别能在自己的工作台完成一天核心工作。
- 普通用户看不到底层四类管理页面。
- 案例候选不会自动公开；只有有效客户展示授权且经过管理员审核才能发布、下架和归档。
- 除 Logo 和小程序展示码外，所有图片上传均被前端和接口拒绝。

### G7：允许正式切换

- 空库安装、现网升级、备份恢复和前端回退全部至少成功演练一次。
- 4.0 空白基线能够在全新项目创建首位管理员、关闭初始化入口并启用 Cron/工作器；与现网升级后的最终结构一致。
- 安全顾问 `ERROR=0`；所有新增对象通过显式授权、RLS、策略、高权限函数执行权和越权测试。
- 金额和库存必须守恒；客户、订单、任务等记录按批准的迁移映射对账，分别列明保留、合并、归档、删除和自动新增数量，不要求无意义的总行数相等。
- 1000 门店、100000 工作项测试通过。
- 空白源码包确认不含原始数据和密钥。
- 老板完成真实业务演练并明确授权切换。

## 14. 必测业务场景

1. 销售在本人区域创建新店线索并完整成交。
2. 销售为其他区域创建线索，进入对应公海或对应销售。
3. 误建线索在无关联记录时删除。
4. 放弃线索进入公海，20 天无人领取后清理。
5. 老客户介绍直接跳到商机或报价，系统保留跳步原因。
6. 竞品老店沉淀，到期 60/30/15 天自动前置。
7. 到期日未知老店不会永久沉底。
8. 同一品牌多个直营门店共用订单并分别交付。
9. 加盟门店独立归属、报价和订单。
10. 报价创建、改版、过期、特价审批和订单冻结。
11. 报价阶段库存不变化；订单付款后正确预留和出库。
12. 软件、硬件、服务在同一订单分别履约。
13. 销售收款、公司收款和混合收款正确分配内部应付。
14. 销售补足内部款后才能启动履约。
15. 服务执行人完成验收后形成劳动收益。
16. 退款分别覆盖销售责任、公司责任和未履约取消。
17. 主管关闭时管理员完成所有审批；开启后按范围路由主管。
18. 离职人员账号停用并完成客户、订单和任务转移。
19. 成交履约后生成案例候选，未审核前网站不可见。
20. 案例上传 Logo 和展示码成功，第三张图、超限和非法格式失败。
21. 所有岗位尝试读取无权收入、客户或财务数据时被拒绝。
22. 新安装空白系统只出现基础字典，不出现任何原公司数据。
23. 收款、出库、验收等业务动作任务无法用通用“完成”绕过业务事务。
24. 会议、拜访和个人日程与工作项共同显示，但不会生成重复任务。
25. 多门店快速建单自动生成报价快照，订单行门店分配数量守恒，每家门店独立履约。
26. 无客户展示授权的案例无法公开；授权撤回后公开 API 和图片副本立即下线。
27. 现网升级链和 4.0 空白基线链最终数据库结构一致，空白链没有历史公司种子。

## 15. 分支、评审和合并制度

- 监理维护 4.0 集成分支，不直接在 `master` 上开发。
- 每个阶段三个团队分别使用独立 `codex/` 分支和独立 worktree。
- 每个团队只改任务单授权目录；跨团队接口先由监理冻结。
- 团队提交后先提供自测证据，监理再交叉审查，不允许团队自行合并到集成分支。
- 每个阶段通过门禁后形成一个可回退检查点，再开始下一阶段。
- 每完成两个正式任务报告一次；每三次正式报告或发生重大业务/迁移变化时压缩一次项目记忆。
- 页面完成、静态检查、候选数据库通过、现网兼容、老板验收、允许合并是六个不同状态，不得混报为“已完成”。

## 16. 上线与回退

1. 施工期间当前 3.0 正常运行；4.0 使用独立预览域名、独立构建工作流和独立测试 Supabase，禁止同一前端构建同时混用测试库和生产库。
2. 预览构建必须校验 Supabase 项目标识属于测试环境，生产构建必须校验属于生产环境；测试数据不得回写生产。
3. 上线前导出数据库、Auth 用户和身份关系、允许图片、Storage 对象清单、表行数、关键金额和库存清单，同时备份 Edge Functions、Cron 计划、功能开关、环境变量名称、提交号和前端产物；密钥只登记安全托管位置，不写入文档和仓库。
4. 先部署兼容数据库迁移，再部署 4.0 前端；数据库迁移不得要求旧前端立即失效。
5. 切换窗口预计 30–60 分钟。若未建设并验收完整双写/增量捕获，则暂停所有可能被迁移、回填或生成关联事件的业务写入，包括线索、客户、订单、任务、财务、库存、履约和案例。
6. 切换后 3.0 页面保留 30 天只读查档入口。
7. 回退分两种：切换完成但尚未产生 4.0 新业务写入时，可由监理核对后恢复上一版 3.0 前端和写入；一旦已经产生 4.0 新写入，3.0 只能继续只读，必须关闭 4.0 写入并向前修复，禁止盲目恢复旧写入口破坏新账本。
8. 数据库只使用向前兼容结构，不在紧急回退中删除新表或新字段。
9. 正式迁移、部署或恢复第一次失败即停止，保存日志、截图、版本和数据清单，等待新的处理决定。

恢复能力最低目标：日常备份的数据损失窗口不超过 24 小时，普通故障恢复时间不超过 4 小时；正式切换前的专项备份应做到接近零业务数据损失。数据库备份和 Storage 文件备份必须分别演练。

切换后的 30 天内每日核对收款、内部应付、利润、库存、积分和员工收益合计，并观察失败任务、越权拒绝、慢查询、主管回落管理员和旧页面使用情况。

## 17. 源码交付包

交付给其他代理商的是单公司空白源码包，包含：

- 两条明确分离的数据库链：现网从 3.0 升级到 4.0 的增量迁移链，以及新客户从零安装的 4.0 空白基线链；

- 部署、升级、备份和恢复手册；
- 私有商业授权文件、第三方开源 LICENSE/NOTICE 和版本清单；

交付包不得包含：员工、客户、品牌、门店、联系人、线索、商机、报价、订单、收付款、工资、利润、操作日志、案例、图片、认证用户和任何访问密钥。演示数据必须另包并默认不安装。

现网升级链继续保留并追加现有 69 个历史迁移，不改写历史。新客户安装器不执行包含 `CANWIN_TEAM`、翻身小队、历史管理员或演示数据的旧种子；它使用从验收后 4.0 结构生成并审查的干净基线，参数化公司名称、首位管理员、项目配置和访问地址。以后每个正式版本同时验证“现网升级”和“全新空库安装”，两条链最终结构必须一致。

## 18. 托管和容量结论

- 当前前端约 1.36MB，GitHub Pages 容量足够继续用于内部试用和 4.0 验收。
- GitHub Pages 只承载静态前端，不承载数据库、任务调度和服务端事务。
- 4.0 正式长期使用前应迁移到适合业务系统的正式静态托管；源码客户自行选择其托管环境。
- 当前 Supabase 数据容量足够，但免费项目存在低活跃暂停和无自动备份风险。
- 正式生产至少执行每日异地数据库导出和允许图片备份；条件允许时升级到带自动备份的付费方案。

## 20. 性能验收指标

最低压测数据集：1000 家门店、50000 条线索/跟进、10000 笔订单、100000 条推进事项、100000 条账本/审计事件，并模拟 30 名活跃用户。

- 常用列表、筛选和搜索 P95 不超过 2 秒。
- 收款、领取、派单和出库等关键写操作 P95 不超过 3 秒。
- 工作台首次业务数据完成不超过 3 秒。
- 深分页不得随页码明显线性变慢，列表必须使用服务端筛选和游标分页。
- 并发领取、收款、派单和扣库存不得产生重复记录或金额/库存不守恒。
- 性能报告必须包含查询计划、响应分位数和错误率，不能只凭页面体感判断。

## 21. 开工判定

本方案已经具备分工、依赖、阶段、验收、迁移、回退和交付边界。正式施工只允许从 P0 开始，P0 未通过不得提前开发 P3–P6 的业务功能。

首批施工产物固定为：

1. 现有数据库对象四分类清单；
2. 角色权限矩阵；
3. 当前页面保留/合并/下线清单；
