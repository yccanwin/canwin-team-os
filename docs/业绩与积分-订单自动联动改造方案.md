# CanWin Team OS 3.0「业绩与积分」订单自动联动改造方案

## 1. 改造目标

将“业绩与积分”改为订单驱动的销售经营页面，淘汰手工积分和手工实际业绩，融合管理看板中有价值的销售管理能力。

最终口径：

- 财务确认客户全款后，订单才计入销售业绩和积分。
- 订单积分就是客如云积分，不保留“预计/官方”双积分。
- 新签/续费在报价时明确选择，并冻结到订单。
- GMV、积分、产品贡献均读取订单及报价快照；商品目录后续改价、改积分不影响历史订单。
- 净收款低于订单总额时，整单业绩和积分撤回；补齐后自动恢复。
- 季度目标由主管设置，实际完成值只能由订单自动产生。

## 2. 页面与菜单方案

保留菜单名称“业绩与积分”和现有页面的视觉风格，删除独立的“管理看板”菜单，将页面整合为五个区域：

1. **季度总览**：积分、新签 GMV、续费 GMV、综合达成率；支持季度切换。
2. **个人/团队业绩**：销售默认看自己；主管可选择销售人员或查看团队排行。
3. **产品贡献**：按订单快照统计产品/套餐销量、积分贡献和 GMV 贡献 TOP5。
4. **季度目标**：主管设置积分、新签 GMV、续费 GMV目标；实际值只读。
5. **销售会议**：融合管理看板的逾期、阻塞、已报价且 7 天内决策商机，并保留负责人、期限和处置记录。

删除或迁移：

- 删除“客如云积分产品”“快速计分”“积分明细”“我的本月观察”。
- 删除“官方积分对账”前端入口；相关旧表暂不删除，避免破坏历史兼容。
- 双账本利润及季度返利/费用移回财务中心，不进入销售业绩页面。
- 下属订单价差仅主管可见，作为团队业绩的权限区块。
- 点击积分或 GMV 卡片可打开“贡献订单”抽屉，用于核对来源；不再设置常驻积分明细模块。

## 3. 数据联动方案

### 3.1 报价与订单

- 报价新增 `sale_type`：`new`（新签）或 `renewal`（续费）。
- 续费报价允许关联原订单；历史客户没有原订单时，保留门店关联并记录“历史续费”。
- 创建订单时冻结销售类型、销售负责人、客户总额、产品/套餐名称、数量和积分。
- 现有历史报价与订单默认标记为新签，不自动计入，只有满足全款条件的订单才回填业绩资格。

### 3.2 业绩资格与冲销

- 净收款 = 已确认客户付款 − 已确认退款冲销。
- 当净收款首次达到订单客户总额时，订单进入“已计业绩”。
- 当净收款跌破订单客户总额或订单取消时，整单撤回。
- 再次补齐全款时自动恢复；重复请求不得重复记账。
- 业绩归属首次全款确认所在季度；后续撤回或恢复均调整该订单原归属季度。
- GMV 取订单客户总额；积分取 `Σ(报价行冻结积分 × 数量)`。

### 3.3 服务端接口与权限

- 新增服务端业绩状态和追加式事件记录，保存“计入、撤回、恢复”的原因、时间和触发单据。
- 付款确认、退款冲销、订单取消必须在同一服务端事务内重算业绩资格。
- 新增统一业绩查询 RPC，返回季度指标、销售排行、产品贡献和贡献订单；前端不自行拼接财务表。
- 销售只能查看本人；主管只能查看负责的销售；老板/管理员按现有权限查看团队；财务权限不自动扩大到销售管理。
- 所有新增表启用 RLS，函数收回 `PUBLIC/anon` 执行权，仅授予 `authenticated`，并保留审计记录。

## 4. 三团队施工分工（共 20 项）

每完成 2 项即为 10% 进度检查点，由总监汇报一次；未通过验收的任务不计入进度。

### 团队 A：订单与业绩数据链（A1–A8）

1. A1：只读核对 V2 商品、套餐、积分和发布条件。
2. A2：确认并发布 V2，验证报价可选择 1 个套餐和 8 个商品。
3. A3：给报价和订单增加新签/续费冻结字段及兼容回填。
4. A4：实现全款业绩状态与追加式事件账本。
5. A5：将付款确认接入自动计入逻辑。
6. A6：将退款、取消和补款接入撤回/恢复逻辑。
7. A7：实现季度业绩、排行、产品贡献和订单来源 RPC。
8. A8：完成 RLS、授权、幂等、审计及数据库回归 SQL。

### 团队 B：业绩与积分统一页面（B1–B7）

9. B1：报价页面增加新签/续费选择和续费来源交互。
10. B2：重构季度总览卡片为订单自动数据。
11. B3：实现个人/团队切换、销售筛选、排行和权限空态。
12. B4：实现产品/套餐贡献 TOP5。
13. B5：保留季度目标编辑，移除所有手工实际值输入。
14. B6：实现贡献订单抽屉及退款/撤回状态说明。
15. B7：完成移动端布局、加载、空数据和错误提示。

### 团队 C：管理看板融合与兼容清理（C1–C3）

16. C1：将逾期、阻塞、临门商机和处置流程融合为“销售会议”。
17. C2：迁移主管订单价差；移除官方对账、月观察和利润区块。
18. C3：删除独立管理看板菜单；旧手工积分数据只保留数据库，不参与新统计。

### 总监：集成、验收与发布（D1–D2）

19. D1：联调新签、续费、全款、退款、恢复、权限和移动端关键路径。
20. D2：受控迁移、生产核对、远端提交、GitHub Pages 发布及线上点击验收。

## 5. 验收标准

- 未全款订单的积分和 GMV 均为 0；全款后只计入一次。
- 部分退款导致净收款不足时整单撤回，补齐后恢复且无重复记录。
- 新签与续费分别进入正确指标；续费能追溯门店及原订单（存在时）。
- 积分等于订单报价快照积分，不受后来商品目录调整影响。
- 销售只能看到本人；主管选择范围内只有销售岗位成员。
- 产品贡献、排行、目标达成和贡献订单的合计互相一致。
- “客如云积分产品、快速计分、积分明细、月观察、官方对账、双账本利润”不再出现在业绩页面。
- 原管理看板菜单消失，销售会议功能在业绩页面可正常处置。
- 手机端可完成季度查看、个人数据查看和销售会议核心操作。

## 6. 发布与停损规则

- 分两批上线：第一批为目录、订单字段和自动业绩数据链；第二批为页面融合和导航清理。
- 数据库迁移前先做只读预检和关键行数记录，不删除旧表、不覆盖历史财务数据。
- 每批只执行一次正式迁移和一次发布尝试；失败立即暂停并汇报，不循环重试。
- 发布后分别核对订单、付款、退款、业绩事件、季度汇总和页面数据。
- 只有生产点击验收通过才记为 100%；构建成功或发布成功不等于业务验收完成。

## 7. 明确不做

- 不重新引入手工积分产品和快速记分。
- 不让报价占用库存。
- 不删除旧积分、官方对账或月观察表，仅停止新写入和前端展示。
- 不把公司利润明细暴露给普通销售或主管。
- 本阶段不做 AI 预测、复杂提成计算和跨销售拆单分佣。
